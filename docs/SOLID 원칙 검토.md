#9 SOLID 원칙 검토

# 9-1 SRP-단일 책임 원칙
|항목|내용|
|---|-----------|
|준수 여부|일부 준수 (설계 고도화를 통해 주요 서비스는 분리했으나, 관리자 기능에 개선 여지 있음)|
|적용 내용|PlayerService 클래스는 선수 비교 대상 지정 및 출력 제어라는 핵심 비즈니스 로직만 담당합니다. 초기 설계에서 발생할 수 있었던 '표/그래프 렌더링'이라는 UI 출력 책임과 '파일 저장'이라는 물리적 포맷 처리 책임을 각각 DisplayStrategy 인터페이스와 ExportService로 완벽히 격리하여 클래스의 결합도를 낮추고 비즈니스 로직 변경의 이유를 하나로 축소했습니다.|
|위반 내용 (해당시)|현재 Admin 클래스는 시스템 인프라 관리(manageSystem)와 데이터 필터링 최적화 로직(optimizeFilterSettings)이라는 두 가지 이질적인 목적의 메서드를 동시에 가지고 있습니다. 시스템 보안/인프라 변경 요인과 스포츠 데이터 분석 필터링 정책 변경 요인은 별개의 도메인이므로, 두 조직의 요구사항이 겹칠 때 Admin 클래스가 동시에 수정되어야 하는 SRP 위반 소지가 존재합니다.|
|개선 방향 (해당시)|Admin 클래스에서 데이터 필터 튜닝 책임을 분리해야 합니다. 시스템 관리는 Admin이 그대로 담당하되, 필터링 데이터 구조 최적화 책임은 FilterManager를 직접 제어하는 독립된 FilterPolicyConfigurator 클래스(가칭)를 신설하여 역할을 위임하는 구조로 개선할 것을 권장합니다.|
# 9-2 OCP-개방-폐쇄 원칙
|항목|내용|
|---|-----------|
|준수 여부|준수 (전략 패턴 도입을 통해 확장 구조 확보)|
|적용 내용|화면 표시 방식(DisplayStrategy)과 이미지 저장 포맷(ExportStrategy)을 구체 클래스가 아닌 인터페이스 뒤로 캡슐화했습니다. 이를 통해 향후 새로운 비교 뷰(예: RadarChartDisplayView)나 새로운 저장 포맷(예: SvgExportStrategy) 요구사항이 추가되더라도, 기존의 호출부인 PlayerService나 ExportService 코드를 단 한 줄도 수정하지 않고 새로운 클래스를 구현하여 주입하는 것만으로 기능 확장이 가능합니다.|
|위반 내용 (해당시)|특이사항 없음. 기능 수정에는 닫혀 있고 새로운 전략 추가(확장)에는 열려 있는 OCP 상태를 완전하게 유지하고 있습니다.|
|개선 방향 (해당시)|현재 구조로도 확장성이 우수하나, 전략 객체들을 동적으로 주입해 줄 수 있는 팩토리 클래스나 의존성 주입(DI) 방식을 코드로 연계한다면 동적 확장성을 더욱 극대화할 수 있습니다.|
# 9-3 LSP-리스코프 치환 원칙
|항목|내용|
|---|-----------|
|준수 여부|준수 (인터페이스 규약 및 부모-자식 간 행위 일관성 유지)|
|적용 내용|DisplayStrategy를 구현하는 TableDisplayView와 GraphDisplayView는 부모 인터페이스의 계약 조건인 renderView()와 clearView()를 예외 던지기나 거부 반응 없이 온전히 오버라이딩하고 있습니다. 호출자인 PlayerService 입장에서는 주입되는 인스턴스가 무엇이든 상관없이 동일한 행위를 기대하고 투명하게 대체하여 사용할 수 있습니다.|
|위반 내용 (해당시)|User를 상속받은 Admin 클래스는 현재 다이어그램 상으로는 위반이 없으나, 만약 실제 구현 단계에서 시스템 관리 기능(manageSystem)을 위해 일반 User 객체가 들어갈 자리에 Admin 객체가 들어왔을 때 런타임 권한 에러를 발생시키거나 instanceof를 통한 타입 다운캐스팅 분기문이 발생한다면 잠재적 LSP 위반으로 이어질 수 있습니다.|
|개선 방향 (해당시)|User와 Admin 간의 상속(IS-A) 관계가 무거워질 경우, 상속을 과감히 제거하고 User 클래스 내부에 권한이나 역할을 나타내는 Role 인터페이스를 속성으로 포함하는 구성(Composition) 방식으로 변경하여 상속으로 인한 LSP 리스크를 사전 차단하는 것이 안전합니다.|
